# 商城运营后台落地计划

## 目标

将当前 Mock 的“黔山优选”运营页改成可日常使用的单店铺商城后台：首页展示真实经营总览，商城菜单负责商品、分类、订单、会员、Banner 和优惠券管理；后端、权限、分页、导出、审计和交互全部沿用 RuoYi 现有模式。

首期只做当前页面真正需要的业务闭环，不预建多商户、支付流水、退款、售后、部分发货、购物车等尚无真实入口的模型。出现真实需求后再通过增量 SQL 扩展。

## 已确认现状

| 项目 | 结论 |
| --- | --- |
| 管理端 | `RuoYi-Vue3/src/views/shopAdmin/index.vue` 是单页 Mock，数据为常量，按钮仅提示成功。 |
| 前端 API | `src/api/shop/index.js` 设置了 `USE_MOCK = true`，只有未接入的 GET 占位接口。 |
| 路由 | 只有 `/shop-admin/overview`，商品、订单等通过同页 Tab 伪装成模块。 |
| 后端 | 没有商城 Controller、Service、Mapper 或领域对象。 |
| 数据库 | 当前 `ry_vue` 只有 RuoYi 和 Quartz 表，没有商城表。 |
| 可复用能力 | RuoYi 已有 CRUD、分页、权限、字典、日志、导出、代码生成和 Quartz，无需增加框架或依赖。 |

## 最小数据模型

### 约束

1. 当前 MySQL 8.0，商城表使用 InnoDB、`utf8mb4`、`datetime(3)`。
2. 内部主键使用自增 `bigint unsigned`；订单号等外部业务号使用不可预测的 `varchar(32)` 唯一值。
3. 金额统一用 `bigint` 分存储，Java 使用 `long`/`BigDecimal` 换算，禁止用 `double` 计算金额。
4. 商品、分类、Banner、优惠券等主数据允许逻辑删除；订单、订单明细、库存流水和订单日志不得删除。
5. `sys_user` 是后台管理员，`mall_member` 是消费者，两者不复用、不建立外键。
6. 订单保存商品、价格、优惠和地址快照，历史订单不受商品改名、改价和地址修改影响。
7. 商品售罄由 SKU 可用库存派生，不保存“售罄”状态。商品只保存草稿、上架、下架。
8. 手机号和地址默认脱敏；查看明文使用单独权限，日志不复制完整敏感信息。

### 表关系

```text
mall_category 1 ── n mall_product 1 ── n mall_product_sku 1 ── n mall_inventory_log

mall_member 1 ── n mall_member_address
     │
     ├── n mall_order 1 ── n mall_order_item
     │         └── n mall_order_log
     └── n mall_member_coupon n ── 1 mall_coupon

mall_banner 独立管理首页内容
```

这 12 张表覆盖首期交易闭环与必要审计，其余能力等真实需求出现后再扩展。

### 商品与库存

| 表 | 关键字段 | 责任 |
| --- | --- | --- |
| `mall_category` | `category_id`、可空 `parent_id`、`category_name`、`level`、`sort_num`、`status`、审计字段、`del_flag` | 两级分类；一级 `parent_id = NULL`，有子分类或商品时不能删除。 |
| `mall_product` | `product_id`、`category_id`、`product_name`、`subtitle`、`main_image`、`images_json`、`detail_html`、`spec_schema_json`、`status`、`publish_time`、`version`、审计字段、`del_flag` | SPU 和展示信息。图片与规格定义用 JSON，避免首期增加低价值关联表。 |
| `mall_product_sku` | `sku_id`、`product_id`、唯一 `sku_code`、`sku_name`、`spec_values_json`、`image_url`、`sale_price`、`market_price`、`available_stock`、`locked_stock`、`warning_stock`、`status`、`version`、`del_flag` | 售价和库存唯一事实源；库存采用条件更新，不能出现负数。 |
| `mall_inventory_log` | `inventory_log_id`、`sku_id`、`biz_type`、`biz_no`、可用/锁定库存变更量及变更前后值、操作人、原因、`create_time`；唯一 `(sku_id, biz_type, biz_no)` | 记录入库、盘点、下单锁定、取消释放和支付实扣；只追加，不编辑。 |

SKU 规格暂用 JSON 是有意简化：当前只需按商品编辑和展示规格，不需要跨商品按规格检索。未来出现该查询后再拆规格定义表。

### 会员

| 表 | 关键字段 | 责任 |
| --- | --- | --- |
| `mall_member` | `member_id`、`app_id`、`open_id`、`union_id`、`nickname`、`avatar_url`、`phone`、`status`、`last_login_time`、创建/更新时间；唯一 `(app_id, open_id)` | 小程序消费者；禁用只阻止新操作，不影响历史订单。 |
| `mall_member_address` | `address_id`、`member_id`、收件人、电话、省市区编码/名称、详细地址、`is_default`、`del_flag`、时间字段 | 保存会员地址；订单创建时复制快照，不依赖地址表展示历史订单。 |

### 订单

| 表 | 关键字段 | 责任 |
| --- | --- | --- |
| `mall_order` | `order_id`、唯一 `order_no`、`member_id`、`order_status`；商品/运费/优惠/应付/实付金额；会员券快照；完整地址快照；支付渠道/渠道交易号；物流公司/运单号；买家留言、后台备注、取消原因、`expire_time`、支付/发货/完成/取消时间、`version`、创建/更新时间 | 订单聚合根。首期一单一次支付、一个包裹，支付和物流字段直接放订单，不增加中间表。 |
| `mall_order_item` | `order_item_id`、`order_id`、`product_id`、`sku_id`；商品名、SKU、规格、图片、单价快照；`quantity`、行金额、优惠分摊、应付金额 | 成交明细快照；订单优惠必须完整分摊到明细。 |
| `mall_order_log` | `order_log_id`、`order_id`、`action`、前后状态、`operator_type`、操作人、脱敏说明、`create_time` | 订单状态审计；只追加。HTTP 操作仍由 `sys_oper_log` 记录。 |

订单只使用一个清晰状态机：

```text
PENDING_PAYMENT ──支付成功──> PENDING_SHIPMENT ──发货──> SHIPPED ──确认收货──> COMPLETED
       └──────────取消/超时关闭──────────> CANCELED
```

已支付订单不能直接取消；退款和售后等到真实支付/售后需求出现后再增加独立表和状态，不把它们提前塞进订单状态。

### 营销内容

| 表 | 关键字段 | 责任 |
| --- | --- | --- |
| `mall_banner` | `banner_id`、标题、图片、`jump_type`、`target_value`、`sort_num`、`status`、展示起止时间、审计字段、`del_flag` | 首页 Banner；仅返回启用且处于展示窗口的数据。 |
| `mall_coupon` | `coupon_id`、名称、门槛金额、优惠金额、发放总量、已领数、已用数、每人限领、领取起止时间、固定有效期或领取后有效天数、`status`、`version`、`del_flag` | 首期只做全店满减券，不预建商品/分类适用范围表。 |
| `mall_member_coupon` | `member_coupon_id`、唯一券号、`coupon_id`、`member_id`、券名/门槛/金额快照、`status`、生效/失效时间、锁定订单、使用订单、领取/使用时间、`version` | `AVAILABLE`、`LOCKED`、`USED`、`EXPIRED`、`VOID`；订单创建锁券、取消解锁、支付核销。 |

### 核心事务

1. 创建订单：校验会员、商品、SKU、实时价格和优惠券；按 SKU ID 顺序执行 `available_stock >= quantity` 条件更新，将库存从可用转为锁定；锁券；写订单、明细、库存流水和订单日志后一次提交，任一步失败全部回滚。
2. 优惠分摊：按符合条件的明细金额比例分摊，尾差由最后一条明细吸收，保证明细优惠合计严格等于订单优惠。
3. 支付成功：以渠道交易号和订单旧状态保证幂等；同一事务将订单改为待发货、`locked_stock` 转为实扣、会员券改为已使用并写流水。重复回调返回原成功结果。
4. 取消/超时：仅待支付订单可执行；同一事务释放库存与会员券并写日志。复用 RuoYi Quartz 扫描 `expire_time`，任务重复运行也只能关闭一次。
5. 发货：仅待发货订单可填写物流信息并变为已发货；同一订单不能重复发货。首期不支持部分发货。

### 索引和数据库约束

1. 使用 `CHECK` 保证金额、数量、库存、优惠券计数和版本号非负；后端保留相同校验以返回可读错误。
2. 建立分类 `(parent_id, status, sort_num)`，商品 `(category_id, status, del_flag)`，SKU 唯一 `sku_code` 与 `(product_id, status)`，会员唯一 `(app_id, open_id)`，订单唯一 `order_no`、唯一 `(pay_channel, channel_trade_no)` 以及 `(member_id, create_time)`、`(order_status, create_time)`，订单明细 `(order_id)`，库存流水唯一 `(sku_id, biz_type, biz_no)`，会员券 `(member_id, status, valid_end_time)` 与 `(coupon_id, member_id)`。
3. 强一致父子关系使用外键 `RESTRICT`，交易表禁止级联删除；逻辑删除后的业务关联由 Service 再校验。
4. 不为单独的低选择性状态列建无效索引，复合索引必须对应页面查询或超时任务。

## SQL 迁移规则

执行任何数据库 DDL 或 DML 前，必须先在 `sql/update/` 生成并检查完整 SQL 文件，禁止执行未落盘的内联 SQL。本期按顺序生成：

1. `2026-07-24-01-新增商城业务表.sql`：12 张表、约束和索引。
2. `2026-07-24-02-新增商城菜单字典.sql`：商城菜单、按钮权限和状态字典。

脚本不得包含 `DROP TABLE`、清表或覆盖式 DML。菜单和字典使用防重复插入；MySQL DDL 中断后先核验已完成对象再续跑，不通过删除已建表重来。初始化不插入假订单、假用户或假成交额，首页允许真实空态。

## 实施顺序

### 1. 数据库和权限

先写并检查两个 SQL 文件，再执行建表、索引、菜单和字典。字典覆盖商品、SKU、库存业务、订单、会员、Banner、优惠券和会员券状态。菜单按钮至少区分查询、新增、修改、删除、导出、库存调整、上下架、发货和敏感信息查看。

### 2. 首页与菜单

经营总览直接放入现有 `/index`，调用 `GET /shop/dashboard` 展示今日已支付订单数、成交额、待发货、新增会员、销售趋势、热销 SKU 和待办。首页不新增商城总览菜单；没有 `shop:dashboard:query` 权限时隐藏经营区，不影响首页其他内容。

商城菜单拆为 `/shop/product`、`/shop/category`、`/shop/order`、`/shop/member`、`/shop/banner` 和 `/shop/coupon`。迁移完成后删除原 `/shop-admin/overview` 路由和 Mock 页面。

### 3. 后端

沿用当前单体结构，不新增 Maven 模块：

1. 在 `ruoyi-system` 增加商城领域对象、DTO、Mapper、Mapper XML 和 Service；在 `ruoyi-admin` 增加 `/shop/...` Controller。
2. 分类、Banner 等机械 CRUD 可用 RuoYi 代码生成器生成后删减；订单、库存和优惠券事务手写，不暴露通用状态/库存更新接口。
3. 列表使用 `startPage()` / `getDataTable()`，详情与写操作使用 `AjaxResult`；写操作使用 `@Transactional(rollbackFor = Exception.class)`、`@PreAuthorize` 和 `@Log`。
4. `ruoyi-quartz` 增加一个薄维护任务，调用 Service 关闭超时订单并过期未使用优惠券，不引入新的调度体系。
5. Controller 不拼状态、不计算金额、不直接组合多表写入，所有业务不变量由 Service 保证。

### 4. 前端

复用 RuoYi 的 `app-container`、查询表单、`right-toolbar`、`pagination`、`v-hasPermi`、字典、确认弹窗和导出，不另造表格或弹窗体系。

商品页完成分类筛选、分页、创建/编辑、SKU 编辑、图片上传、上下架和库存调整；订单页完成筛选、详情、取消待支付订单和发货；会员页完成查询、地址只读查看和启停；Banner、优惠券完成各自 CRUD、启停与有效期校验。所有页面必须覆盖加载、空数据、失败、无权限和提交中状态。

## 业务规则

| 领域 | 最小但不可省的规则 |
| --- | --- |
| 商品 | 至少一个启用 SKU、分类启用、价格和库存合法才能上架；售罄由 SKU 可用库存派生；逻辑删除不影响订单快照。 |
| 分类 | 固定两级；停用分类不可分配新商品；有子分类或商品时不能删除。 |
| 库存 | 新建 SKU 的库存固定为 0，初始库存也通过入库命令登记；人工入库/盘点每次必须写原因和流水；下单锁定、取消释放、支付实扣不能走通用编辑接口。 |
| 订单 | 后端计算全部金额；待支付才能取消，待发货才能发货，已发货才能完成；状态更新必须带旧状态或版本条件。 |
| 优惠券 | 同时校验总量、每人限领、领取窗口、有效期和订单门槛；订单创建锁券，取消解锁，支付核销。 |
| Banner | 只返回启用且处于展示窗口的数据；跳转目标保存时校验。 |

首页统计口径固定为：今日订单数按支付成功时间统计；成交额为实付金额；待发货为 `PENDING_SHIPMENT`；新增会员按创建时间统计；销量为已支付商品数量。日区间使用 `Asia/Shanghai` 左闭右开区间，前端不得对当前页列表求和代替聚合接口。

## 验证与验收

1. SQL 在空库副本执行成功，重复执行不产生重复菜单/字典；实际表、列、约束和索引与计划一致。
2. 商品、分类、会员、Banner、优惠券 CRUD、分页、筛选、权限与导出正常。
3. 多 SKU 下单全部成功或全部回滚；库存不足、重复请求、取消与支付竞争不产生负库存、重复流水或重复核销。
4. 订单状态所有允许和拒绝的跃迁有测试；重复支付回调、重复取消、重复发货幂等或被明确拒绝。
5. 优惠券总量、限领、锁定、解锁、核销和过期一致；订单优惠与明细分摊金额严格相等。
6. 首页统计与相同条件的订单明细一致；首页和管理页覆盖加载、空态、失败和无权限状态。
7. 后端测试、Maven 构建、前端构建和关键交互回归全部通过，没有 Mock 数据或“预留成功”提示。

## 后续按需扩展

以下能力不在首期预建表：购物车持久化、支付尝试流水、退款、售后、部分发货、多仓、商品规格检索、优惠券商品范围、多商户、分账、WMS 和第三方物流。真实需求出现时，先在 `sql/update/` 写增量 SQL，再在现有订单/SKU/优惠券边界上扩展，不推翻首期模型。
